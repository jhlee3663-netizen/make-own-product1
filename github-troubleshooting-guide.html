Background
무엇을 하려고 했나?

개인 GitHub 계정(jhlee3663-netizen)에서 작업한 React 앱을 팀 조직 레포지토리(BF-D-challenge/make-own-product1)에 업로드하려 했습니다.

  • 개인 레포(jhlee3663-netizen/make-own-product1) 생성 & 코드 Push: 성공
  • 팀 레포 Write 권한 획득 (팀장에게 요청): 성공
  • GitHub Actions 워크플로우 설정: 시도 (→ 실패)
  • 터미널에서 직접 Push: 최종 성공 🎉
⏱ 삽질 타임라인
❌
시도 1 — 실패
GitHub Actions 자동 동기화 (최초 시도)

개인 레포에서 팀 레포로 자동으로 코드를 동기화하는 GitHub Actions 워크플로우(.github/workflows/main.yml)를 만들었습니다.

Process completed with exit code 128. remote: Permission to BF-D-challenge/make-own-product1.git denied to github-actions[bot]. fatal: unable to access '...': The requested URL returned error: 403
🔍 실패 원인 GitHub Actions 봇(github-actions[bot])이 팀 레포에 접근하려면 별도의 토큰이 필요한데, ORG_TOKEN Secret이 올바르게 설정되지 않아 기본 봇 계정으로 접근을 시도한 것. 당연히 권한 없음으로 차단됨.
🔑
시도 2 — 실패
ORG_TOKEN Secret 설정 후 재시도

GitHub Personal Access Token(PAT)을 발급하고 개인 레포의 Settings → Secrets → ORG_TOKEN에 등록 후 Actions를 재실행했습니다.

remote: Permission to BF-D-challenge/make-own-product1.git denied to github-actions[bot].
🔍 실패 원인 같은 에러 반복. 워크플로우 yml 파일에서 토큰을 URL에 올바르게 삽입하지 못해 여전히 github-actions[bot] 계정으로 인증 시도. 토큰이 있어도 yml 코드가 잘못되면 의미가 없음.
🔐
시도 3 — 부분 성공 후 실패
Write 권한 획득 후 재시도

팀장에게 BF-D-challenge/make-own-product1에 대한 Write 권한을 요청하여 받았습니다. 권한을 받았으니 당연히 될 것이라 생각했지만…

Process completed with exit code 128. (여전히 동일한 에러)
🔍 실패 원인 팀 레포의 main 브랜치에 Branch Protection Rule(브랜치 보호 규칙)이 설정되어 있었음. Write 권한이 있어도 봇(Actions)이 직접 main을 수정하는 건 막혀있는 구조. 또한 워크플로우의 토큰 인식 자체도 여전히 불안정한 상태.
🔄
시도 4 — 실패
토큰 재발급 + 워크플로우 코드 수정

PAT를 새로 발급(repo + workflow 권한)하고, 워크플로우 yml 파일을 x-access-token 방식으로 수정 후 재시도했습니다.

# 수정한 워크플로우 핵심 부분 env: TOKEN: ${{ secrets.ORG_TOKEN }} run: | git remote add org-repo https://x-access-token:${TOKEN}@github.com/BF-D-challenge/make-own-product1.git git push org-repo HEAD:main --force
remote: Permission to BF-D-challenge/make-own-product1.git denied to github-actions[bot].
🔍 최종 진단 GitHub Actions 환경에서 조직 레포로 PAT를 이용해 Push하는 방식은 조직의 보안 정책 및 브랜치 보호 규칙에 의해 막힐 가능성이 높음. GitHub Actions 방식 자체를 포기하고 터미널 직접 Push로 전환 결정.
🚀
시도 5 — 최종 성공! 🎉
터미널에서 직접 Push (토큰 인증)

GitHub Actions를 완전히 포기하고, 로컬 터미널에서 토큰을 URL에 직접 포함시켜 팀 레포를 remote로 추가한 뒤 직접 push 했습니다.

# 1. 팀 레포를 'org' 이름으로 remote 등록 (토큰 포함) git remote add org https://내계정명:ghp_토큰값@github.com/BF-D-challenge/make-own-product1.git # 2. 팀 레포에 Push! git push org main --force
✅ 결과: To https://github.com/BF-D-challenge/make-own-product1.git
+ 31b40e3...e95baf9 main → main (forced update)
팀 레포에 코드가 완벽히 올라갔습니다!
🧠 왜 이 방법만 됐나?

GitHub Actions 봇과 직접 터미널 Push의 핵심 차이점:


❌ GitHub Actions (실패)

깃허브 서버가 대신 실행. 조직의 보안 규칙(branch protection, PAT 제한)에 영향을 많이 받음. 토큰 인식도 불안정할 수 있음.

✅ 터미널 직접 Push (성공)

내 컴퓨터에서 직접 실행. 팀장이 부여한 Write 권한 + 내 계정 토큰으로 인증하므로 조직이 명확히 "내가 한 것"으로 인식함.

📋 이후 사용법 (복붙 가능)
💻
앞으로 팀 레포에 Push할 때
딱 이 순서대로!

org remote는 한 번만 등록하면 됩니다. 이미 됐다면 1번은 건너뛰세요.

# ① 처음 한 번만: 팀 레포 remote 등록 git remote add org https://내계정명:ghp_토큰@github.com/BF-D-challenge/make-own-product1.git # ② 코드 수정 후 staging & commit git add . git commit -m "수정 내용 설명" # ③ 내 개인 레포에 Push git push origin main # ④ 팀 레포에 Push git push org main
💡 꿀팁 & 주의사항
🔑

토큰은 한 번만 보여줌

PAT(Personal Access Token)는 생성 직후에만 볼 수 있습니다. 반드시 메모장이나 안전한 곳에 저장해두세요!

⏰

토큰 만료 주의

토큰에 만료일을 설정했다면, 만료 후 git push org main이 안 됩니다. 이때는 토큰 새로 발급 후 remote URL을 업데이트하세요.

🛡️

토큰을 코드에 절대 넣지 말 것

토큰은 remote URL에만 사용하고, 소스 코드 파일에 직접 넣으면 보안 사고의 원인이 됩니다. GitHub에 올라가는 순간 탈취 위험!

👮

Write 권한은 팀장에게 요청

조직 레포에 Push하려면 관리자에게 Collaborator Write 권한을 받아야 합니다. 권한 없이는 아무리 해도 403 에러!

⚡

--force 사용 시 주의

--force는 기존 팀 레포 내용을 덮어씁니다. 팀원이 올린 코드가 있다면 날아갈 수 있으니, 팀과 먼저 소통 후 사용하세요!

🌿

브랜치 보호 규칙 확인

팀 레포의 main이 보호되어 있다면, 직접 Push 대신 별도 브랜치로 Push 후 Pull Request를 여는 걸 추천합니다.

🏆
